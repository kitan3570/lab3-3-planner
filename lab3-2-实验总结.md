# Lab 3-2 实验总结

## 1) 前端和后端分别负责了什么？

前端（Vue 3）主要负责“交互与呈现”：
- 提供规划创建表单（标题、日期、预算、人数、偏好、备注），并把用户输入转成请求发送给后端。
- 提供地图选点与参数录入（时段、预计花费、停留时长、备注），并把选点结果加入当前规划。
- 提供行程安排视图（按天、按上午/下午/晚上分组），支持对地点信息的编辑、删除、导出行程单、预算超额提醒等。
- 提供“历史记录”弹窗，展示历史规划列表并支持打开/删除。
- 提供 AI 辅助总结卡片（按钮 + loading + 结果展示），将后端返回的建议内容呈现给用户。

后端（FastAPI + SQLite）主要负责“数据与业务规则”：
- 提供规划与地点的 RESTful API：创建规划、查询规划详情、添加地点、更新地点、删除地点、列出规划摘要、删除规划等。
- 管理持久化数据结构（Plan、Location），并处理多日规划字段 `day_index` 的存储与迁移。
- 作为 BFF（Backend For Frontend）整合第三方能力：
  - 天气：根据地点坐标调用天气服务并把结果附加到返回结构中。
  - AI：根据规划与地点信息组装 prompt，调用大模型接口生成建议文本后返回给前端。
- 负责安全与配置：第三方 key 只保存在后端 `.env` 中，由后端读取并发起请求，前端不接触密钥。

## 2) 你是如何组织规划数据的？

数据结构以“规划（Plan）→ 地点（Location）”为核心：
- 一个 Plan 表示一次出行规划，包含：`title/date/budget/people_count/preferences/remarks` 等基本信息。
- 一个 Plan 拥有多个 Location（一对多），每个地点包含：
  - 位置：`name/lat/lng`
  - 行程属性：`day_index`（第几天，默认 1）、`time_slot`（上午/下午/晚上）
  - 成本与时长：`estimated_cost/duration`
  - 补充说明：`remarks`

前端的“当前规划”以一个 `currentPlan`（对应后端的 PlanRead）作为单一事实来源：
- 当创建规划、添加地点、更新地点、删除地点后，会用后端返回的数据更新 `currentPlan`，必要时再 `refreshPlan()` 拉取最新状态。
- 多日规划通过 `dayCount + selectedDayIndex` 控制视图分层：同一规划下，不同 `day_index` 的地点在 UI 上通过 Tab 隔离展示，但数据仍在同一个 Plan.locations 数组中统一管理。

## 3) 你如何处理天气信息和规划之间的关系？

天气信息被设计成“派生信息”而不是规划的持久化字段：
- Location 的核心数据（时间、花费、时长、备注、day/time_slot 等）需要稳定保存到数据库；天气属于易变化、可失效、且依赖外部服务的动态信息，不适合写入数据库作为事实数据。
- 后端在返回规划详情、或新增/更新地点时，根据 `lat/lng` 调用天气服务并把结果附加到响应的 `weather` 字段中。
- 这样做的好处：
  - 数据一致性更清晰：规划本身不被外部波动污染；天气失败也不会影响规划数据存取。
  - 失败可控：天气服务不可用时，仍可返回规划与地点信息，只是 `weather.ok=false` 并附带错误原因。
  - 前端体验更稳定：前端按 `weather.ok/summary` 进行展示与降级，不阻塞用户编辑与删除地点。

## 4) AI 在系统中的角色是什么，为什么它不是主角？

AI 在系统中扮演“辅助总结与建议生成器”的角色：
- 它读取结构化规划数据（计划基本信息、地点清单、时间分布、成本汇总、天气信息），输出一段更易读、更具行动性的建议文本。
- 它的产出是建议，不会直接修改数据库数据，也不会替代用户做关键决策。

AI 不是主角的原因：
- 规划系统的主干价值来自结构化数据与可控的 CRUD：用户需要可编辑、可追溯、可重复使用的行程组织能力，而不是一次性的生成文本。
- AI 的输出具有不确定性与成本：同样的输入可能产生不同输出；调用速度与费用也不可忽视，因此更适合“按需触发”，而非每次渲染都依赖。
- 安全与隐私约束：密钥必须留在后端，且输入内容应尽量可控；把 AI 放在“后端可控的 BFF 层”更符合安全边界。

## 5) 在这个实验中，你觉得最难的是前端、后端、数据管理，还是前后端协作？为什么？

我认为最难的是前后端协作，其次是数据管理。

原因主要有三点：
- 需求迭代导致数据协议变化频繁：从单日到多日（`day_index`）、从只展示到可编辑（PUT）、从静态列表到“历史记录/模板/导出”等，都会影响 schema、接口、前端状态组织和 UI 交互。
- 一致性与刷新策略需要权衡：例如编辑地点采用自动保存（debounce），删除地点与删除天数涉及批量请求与状态回收，既要让 UI 立刻响应，又要保证最终状态与后端一致。
- 外部依赖的降级与边界：天气与 AI 都可能失败或变慢，因此需要在后端与前端都设计“可用但不完美”的降级路径，避免主流程被外部服务绑架。

从结果看，采用“后端做 BFF、前端做可视化与交互、外部能力做可选增强”的分层方式，让系统在不断加功能时仍能保持结构清晰。
